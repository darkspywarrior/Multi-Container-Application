* Files can be modified.

# What happens in your system (actual flow)

## 1. User uploads a file

Example:
`contract.pdf`

System:

* accepts file
* computes fingerprint (SHA-256/SHA3 etc)
* hash becomes digital fingerprint

Example:

```text
contract.pdf
→ SHA256
→ a84f...91c
```

That hash uniquely represents file content.

If one bit changes:
new hash.

---

# 2. File stored in MinIO

Actual content goes to MinIO object storage.

MinIO stores:

* original file
* object metadata
* versions
* timestamps

MinIO = content layer.

---

# 3. Hash goes to Hyperledger Fabric

You do NOT put file in blockchain.

You put:

```json
{
 assetId,
 fileName,
 hash,
 owner,
 timestamp,
 txId
}
```

Fabric gives:

* endorsement
* consensus
* immutable ledger
* replicated proof across peers

Blockchain = trust layer.

---

# 4. Verification

Later user uploads same file again.

System:

* re-hashes file
* queries blockchain stored hash
* compares

If equal:
VALID

Else:
TAMPERED

That’s your integrity proof.

---

# 5. Chain of custody / audit

Every upload can create:

* versions
* ownership records
* transaction history
* audit trail

That becomes evidentiary record.

# What is “actually happening” in blockchain side

When upload occurs:

1 client submits proposal

2 peers simulate chaincode

3 endorsers sign response

4 orderer orders transaction

5 block formed

6 block hash linked to previous block

7 Merkle root created

8 peers commit block

9 world state updated

10 verification query later checks state

This is why it is tamper resistant.
